# Testing Tier System — Formal Specification

**Version:** 3.0
**Status:** Source of truth
**Supersedes:** Version 2.0 (retained in git history); `TEST_TIER_SYSTEM.md` (informal predecessor, retained for historical context only)

**v3.0 Changes (Structural Achievability Fix):**
- **M_t REMOVED as a gate** (now informational only). M_t measures call-graph structure, not test quality. Orchestrator methods (e.g., Gradle plugin `apply()`) structurally reach 6+ sub-methods. Penalizing this conflates good integration testing with mega-testing.
- **R replaced by R_direct** when source analysis is available. R_direct counts only tests that DIRECTLY call the mutated method, eliminating structural inflation from transitive kills. Falls back to R with lenient thresholds when source analysis is unavailable.
- **M_p thresholds relaxed** to accommodate integration verification patterns where tests legitimately call 2-3 production methods.
- **Coverage thresholds aligned** between spec and implementation (v3.0 code had lower thresholds than v2.0 spec; now unified).
- **Perfection reachability** updated: now achievable for Gradle plugin libraries without production code refactoring (when mutation score = 100%).

## 1. Behavioral Contract

The Testing Tier System is founded on three axioms. A test suite satisfies the contract if and only if all three axioms hold simultaneously. Axiom 3 is enforced by **two** gates: granularity (Gate 3a, primary methods) and direct redundancy (Gate 3b, direct killers per mutation). M_t (transitive methods per test) is reported as an informational diagnostic but is NOT a gate.

### Axiom 1 — Coverage Completeness (JaCoCo)

> **Every instruction and every branch in production code must be exercised by at least one test.**

Formally:

- For every JVM bytecode instruction `i` in the instrumented production codebase, there exists at least one test `t` such that `t` executes `i`.
- For every branch edge `b` (control-flow edge arising from `IF`, `IFELSE`, `SWITCH`, `WHEN`, ternary, or short-circuit operators) in the production codebase, there exists at least one test `t` such that `t` traverses `b`.

**Measurement:** JaCoCo bytecode instrumentation. The JaCoCo agent records, per instruction and per branch edge, whether it was executed during the test run. The aggregate counters are emitted in the JaCoCo XML report.

**Scope:** "Production code" is defined as all classes included in the JaCoCo report after applying the project's JaCoCo exclusion filters. Classes excluded from JaCoCo are out of scope for Axiom 1 but are subject to the Exclusion Ratio metric (Section 2.9).

**Implication:** 100% instruction coverage is necessary but not sufficient for 100% branch coverage. 100% branch coverage is sufficient for 100% instruction coverage. Both are required at the Perfection tier.

### Axiom 2 — Mutation Kill Completeness (PIT)

> **Every mutation generated by PIT must be killed by at least one test.**

Formally:

- Let `M = {m_1, m_2, ..., m_n}` be the set of all mutations generated by PIT for the production codebase under the configured mutators and exclusion filters.
- A mutation `m_i` is **killed** if at least one test fails when `m_i` is applied to the codebase.
- A mutation `m_i` is **alive** if it is not killed (status `SURVIVED`, `NO_COVERAGE`, `TIMED_OUT`, or `RUN_ERROR`).
- Axiom 2 holds if and only if `|{m_i : m_i is killed}| = |M|`, i.e., mutation score = 100%.

**Measurement:** PIT mutation testing. PIT applies each mutation, runs the test suite, and records the outcome. The full mutation matrix (all mutations x all tests) is emitted in the PIT XML report when `fullMutationMatrix=true`.

**Scope:** "Every mutation" is defined as every `<mutation>` element in the PIT XML report. Mutations in classes excluded from PIT are not generated and therefore not counted. Equivalent mutants (mutations that cannot be killed because the mutated code is semantically equivalent to the original) are a known limitation (Section 8.4) and must be excluded from PIT via configuration, not silently tolerated.

**Implication:** Axiom 2 subsumes Axiom 1 in the limit — if every mutation is killed, coverage is necessarily high. However, Axiom 1 is retained because (a) coverage is cheaper to compute, (b) coverage gaps are more actionable than surviving mutations, and (c) the two axioms catch different classes of test deficiencies.

### Axiom 3 — Minimal Waste

> **Axioms 1 and 2 must be achieved with minimum waste, measured by two gates: granularity (Gate 3a, primary methods) and direct redundancy (Gate 3b, direct killers per mutation). M_t (transitive reach) is reported as an informational diagnostic.**

A test suite has **waste** if it contains tests that are:
- **Mega-tests** (Gate 3a): a single test directly exercises too many primary production methods.
- **Redundant tests** (Gate 3b): multiple tests directly call and kill mutations in the same production method.

**Why M_t is no longer a gate (v3.0 change):** M_t counts the number of distinct production methods whose mutations are killed by a test, including methods reached transitively through the call graph. This conflates two fundamentally different scenarios:
1. **Orchestrator integration testing** (M_t = 4-6): A test calls `apply()` which calls `registerConvenienceTasks()` → `resolveCompileTaskNames()`. The test is testing the apply flow's integration. This is GOOD test design, structurally inevitable for orchestrator-heavy codebases like Gradle plugins.
2. **Facade mega-testing** (M_t = 50): A test calls `GodRunner.run()` which calls 50 unrelated methods. This is BAD test design.

M_t cannot distinguish between these. Penalizing scenario 1 forces production code refactoring (extracting orchestrators into pure functions) or impractical mocking (you cannot meaningfully mock `project.tasks.register()`). This violates the principle that the tier system should measure test quality, not production code architecture.

M_p (Gate 3a) already catches genuine mega-tests — a test that directly calls 50 production methods has M_p = 50. The facade mega-test scenario (M_p = 1, M_t = 50) is a legitimate integration test for the facade's entry point, not a mega-test.

#### Gate 3a — Granularity (M_p, primary methods)

> **The maximum number of primary methods exercised by any single test must not exceed M_p_threshold.**

A **primary method** is a production method that is **directly invoked** by the test under test (i.e., the test calls the method directly, not transitively through other production methods). A test that exercises many primary methods is a "mega-test" — it is coarse, hard to maintain, and couples many behaviors to a single test.

**Parameter M_p** is tier-dependent (Section 4). The metric `M_p` (primary methods per test) is defined in Section 2.4.

**Critical clarification:** The value of `M_p` computed from PIT data alone is **transitive** — it counts all methods whose mutations are killed by the test, including methods reached transitively through the call graph. True primary `M_p` requires source-code analysis. The hybrid computation is defined in Section 2.4.3.

#### ~~Gate 3b — Transitivity (M_t, transitive methods)~~ — REMOVED in v3.0

> **M_t is now informational only. It is NOT a gate.**

M_t was removed as a gate in v3.0 because it measures call-graph structure (production code architecture), not test quality. For details on why it was removed, see the Axiom 3 section above. M_t is still computed and reported as a diagnostic metric, but it does not affect tier calculation.

The metric definition (Section 2.5) is retained for the informational diagnostic.

#### Gate 3b — Direct Redundancy (R_direct, direct killers per mutation)

> **The maximum number of tests that BOTH kill a mutation AND directly call the mutated method must not exceed R_direct_threshold.**

R_direct replaces R (all killers) as the redundancy gate in v3.0. The key distinction:
- **R** (old): counts ALL tests that kill a mutation, including tests that kill it transitively (the test's primary method calls the mutated method through the call graph).
- **R_direct** (new): counts ONLY tests that directly call the mutated method as a primary method AND kill its mutations.

**Why R_direct replaces R:** R is structurally inflated by transitive kills. When `testApply()` calls `apply()` which calls `filterExistingDirs()`, `testApply()` transitively kills mutations in `filterExistingDirs()`. But `testApply()` is NOT redundant with `testFilterExistingDirs()` — they test different integration paths. R_direct excludes `testApply()` from the count for `filterExistingDirs()` mutations because `filterExistingDirs` is not a primary method of `testApply()`.

**Fallback when source analysis is unavailable:** If `--test-src` is not provided, R_direct cannot be computed. The system falls back to R (all killers) with lenient thresholds that accommodate structural transitive kill inflation:

| Tier | R_direct (preferred) | R (fallback) |
|------|--------------------|-------------| < 1% |
| Perfection | <= 3 | <= 5 |
| Platinum | <= 3 | <= 6 |
| Gold | <= 4 | <= 8 |
| Silver | <= 6 | <= 12 |
| Bronze | <= 10 | <= 20 |

**Interaction between Gates 3a and 3b:** The two gates are independent. A test suite can have low `M_p` (fine-grained primary calls) but high `R_direct` (many tests directly calling the same method), or high `M_p` (coarse tests) but low `R_direct` (few redundant direct tests). Both must be satisfied.

## 2. Interface Definition

This section defines every metric precisely: its formula, data source, and computation procedure.

### 2.1 Instruction Coverage

**Formula:**

```
instruction_coverage = instruction_covered / (instruction_covered + instruction_missed) * 100
```

**Data source:** JaCoCo XML report (`jacocoTestReport.xml` or equivalent).

**XML path:** Root `<report>` element's direct child `<counter type="INSTRUCTION">`. The root-level counter aggregates all packages, classes, and methods.

**Computation:**

1. Parse the JaCoCo XML.
2. Locate the root-level `<counter type="INSTRUCTION">` element (a direct child of `<report>`, not nested inside a `<package>` or `<class>`).
3. Read `covered` attribute -> `instruction_covered` (integer).
4. Read `missed` attribute -> `instruction_missed` (integer).
5. Compute: `instruction_coverage = instruction_covered / (instruction_covered + instruction_missed) * 100`.
6. If `instruction_covered + instruction_missed == 0`, the metric is `0.0` (no production code instrumented).

**Unit:** Percentage, one decimal place.

**Semantics:** `instruction_covered` is the count of JVM bytecode instructions that were executed by at least one test during the JaCoCo-instrumented test run. `instruction_missed` is the count of instructions that were never executed. JaCoCo counts every bytecode opcode: `ILOAD`, `IADD`, `IFGT`, `ARETURN`, `INVOKEVIRTUAL`, etc.

### 2.2 Branch Coverage

**Formula:**

```
branch_coverage = branch_covered / (branch_covered + branch_missed) * 100
```

**Data source:** JaCoCo XML report (same file as instruction coverage).

**XML path:** Root `<report>` element's direct child `<counter type="BRANCH">`.

**Computation:**

1. Parse the JaCoCo XML.
2. Locate the root-level `<counter type="BRANCH">` element.
3. Read `covered` attribute -> `branch_covered` (integer).
4. Read `missed` attribute -> `branch_missed` (integer).
5. Compute: `branch_coverage = branch_covered / (branch_covered + branch_missed) * 100`.
6. If `branch_covered + branch_missed == 0`, the metric is `0.0` (no branches in production code).

**Unit:** Percentage, one decimal place.

**Semantics:** A branch edge is a control-flow edge arising from a conditional jump instruction (`IFEQ`, `IFNE`, `IFGT`, `IFLT`, `TABLESWITCH`, `LOOKUPSWITCH`, etc.). `branch_covered` counts edges where both directions (true and false) were taken during the test run. `branch_missed` counts edges where at least one direction was never taken. Note: JaCoCo counts branch *edges*, not branch *nodes*. An `if` statement has 2 edges (taken and not-taken); both must be exercised for the branch to be "covered."

**Relationship to instruction coverage:** 100% branch coverage implies 100% instruction coverage (every instruction on every branch edge is executed). The converse is not true: 100% instruction coverage does not imply 100% branch coverage (all instructions on both sides of a branch could be executed without the branch itself being taken in both directions — though this is rare in practice).

### 2.3 Mutation Score

**Formula:**

```
mutation_score = killed_mutations / total_mutations * 100
```

**Data source:** PIT XML report (`mutations.xml` or equivalent).

**XML path:** All `<mutation>` elements under the root element. The status is read from the `status` attribute of each `<mutation>` element.

**Computation:**

1. Parse the PIT XML.
2. Count all `<mutation>` elements -> `total_mutations`.
3. Count `<mutation>` elements with `status="KILLED"` -> `killed_mutations`.
4. Compute: `mutation_score = killed_mutations / total_mutations * 100`.
5. If `total_mutations == 0`, the metric is `0.0` (no mutatable code).

**Unit:** Percentage, one decimal place.

**Semantics:** `total_mutations` is deterministic — it depends only on the mutatable bytecode patterns in the production code and the configured mutator set. It does not depend on the test suite (the same codebase always produces the same mutations). `killed_mutations` is the count of mutations where at least one test failed when the mutation was applied.

**PIT status values:** `KILLED` (test failed), `SURVIVED` (all tests passed), `NO_COVERAGE` (no test exercised the mutated code), `TIMED_OUT` (test exceeded time limit — treated as NOT killed), `RUN_ERROR` (PIT could not run — treated as NOT killed). Only `KILLED` counts toward `killed_mutations`.

### 2.4 M_p — Primary Methods Per Test (Gate 3a: Granularity)

#### 2.4.1 Definition

**M_p(t)** for a test `t` is the number of **primary methods** exercised by `t`.

A **primary method** is a production method that is **directly invoked** by the test `t`. "Directly invoked" means the test's call stack contains a direct call from the test method to the production method, with no intervening production method on the stack. In other words, the production method is a direct callee of the test method, not a transitive callee reached through other production methods.

Formally, let `G` be the production call graph (nodes = methods, edges = calls). Let `T` be the set of test methods. For a test `t` in `T`, the set of primary methods `P(t)` is:

```
P(t) = { m in ProductionMethods : exists edge (t, m) in the combined call graph G union {edges from tests to production} }
```

That is, `m` is a primary method of `t` if and only if `t` calls `m` directly (not through another production method).

The metric **M_p** for the test suite is:

```
M_p = max_{t in T} |P(t)|
```

The gate checks: `M_p <= M_p_threshold` where `M_p_threshold` is the tier threshold.

#### 2.4.2 Why PIT Data Alone Is Insufficient (Transitivity Problem)

PIT's `killingTests` field records, for each killed mutation, which test(s) killed it. From this, we can derive which methods' mutations were killed by each test. However, this gives us the **transitive** set of methods — all methods whose mutations were killed, including methods reached through the call graph. This is the metric `M_t` (Section 2.5), not `M_p`.

Example:
- Test `testCheckout()` calls `Cart.checkout()`, which calls `Inventory.deduct()`, which calls `Database.update()`.
- PIT reports that `testCheckout()` killed mutations in `Cart.checkout()`, `Inventory.deduct()`, and `Database.update()`.
- Transitive method count from PIT = 3 (three methods' mutations were killed). This is `M_t(testCheckout) = 3`.
- Primary method count from source analysis = 1 (only `Cart.checkout()` is directly called by the test). This is `M_p(testCheckout) = 1`.

The transitive count is always >= the primary count. Using transitive count as a proxy for primary count is **conservative** — it over-counts, which means it may flag tests as too coarse when they are actually fine-grained. This is why `M_p` requires source analysis (Section 2.4.3) while `M_t` can be computed from PIT data alone (Section 2.5).

#### 2.4.3 Hybrid Computation of M_p (PIT Fast-Path + Source Fallback)

The computation of `M_p` proceeds in two phases:

**Phase 1 — PIT Fast-Path (Transitive Upper Bound):**

1. Parse the PIT XML with `fullMutationMatrix=true`.
2. For each killed mutation, read the `killingTests` element (pipe-separated list of test method names).
3. For each test `t`, collect the set of methods whose mutations were killed by `t`. A method is identified by the `mutatedClass` and `mutatedMethod` elements of the mutation.
4. Compute `M_t(t) = |{distinct (mutatedClass, mutatedMethod) pairs killed by t}|` (this is the transitive method count, used as informational diagnostic).
5. Compute `M_t = max_t M_t(t)`.
6. If `M_t(t) <= M_p_threshold` for all tests `t`, then Gate 3a is **PASS**. The transitive upper bound is already below the primary threshold, so the true primary `M_p` (which is <= transitive) is also below the threshold. No source analysis is needed.
7. If `M_t(t) > M_p_threshold` for any test `t`, proceed to Phase 2 for that test.

**Phase 2 — Source Analysis (Primary Determination):**

1. For each test `t` where `M_t(t) > M_p_threshold`, perform source-level call analysis:
   - Locate the test source file containing `t`.
   - Parse the test method body.
   - Identify all direct method calls in the test method body (calls to production methods, not calls to test framework methods, assertion methods, or other test utility methods).
   - The set of production methods directly called = `P(t)`.
   - `M_p(t) = |P(t)|`.
2. Compute `M_p = max_t M_p(t)` (only over tests that entered Phase 2; for tests that did not enter Phase 2, `M_p(t) <= M_t(t) <= M_p_threshold`, so they cannot be the maximum).
3. The gate checks: `M_p <= M_p_threshold`.
4. If `M_p <= M_p_threshold`, the gate is **PASS**. The transitive count was inflated by transitive calls, but the primary count is acceptable.
5. If `M_p > M_p_threshold`, the gate is **FAIL**.

**Source analysis method classification:** A method call in a test body is classified as follows:
- **Production method call:** The callee is a method in a production source directory (e.g., `src/main/kotlin/` or `src/main/java/`). These are counted as primary methods.
- **Test framework call:** The callee is from JUnit, TestNG, MockK, Mockito, or another test framework. These are NOT counted.
- **Assertion call:** The callee is an assertion method (`assertEquals`, `assertTrue`, `verify`, etc.). These are NOT counted.
- **Test utility call:** The callee is a method defined in a test source directory (e.g., `src/test/kotlin/`) that is not the test method itself (e.g., a helper method). These are NOT counted.
- **Kotlin stdlib / JDK call:** The callee is from `kotlin.collections`, `kotlin.io`, `java.lang`, `java.util`, etc. These are NOT counted.

**Implementation note:** Source analysis requires parsing Kotlin (or Java) source files. For Kotlin, a lightweight approach using regex or Kotlin compiler PSI (Program Structure Interface) is sufficient — we only need to identify direct method calls in the test method body, not full semantic analysis. The implementation should resolve imports to determine whether a callee is a production method or not. See Section 5.5 for the full source analysis procedure.

#### 2.4.4 When Source Analysis Fails (M_p Indeterminable)

If source analysis cannot determine `M_p(t)` for a test `t` (e.g., the test source file cannot be found, the method body cannot be parsed, or the call classification is ambiguous), the following fallback applies:

1. **Source file not found:** If the test source file cannot be located by searching test source directories for the test method name, `M_p(t)` is **indeterminable**. The conservative fallback is `M_p(t) = M_t(t)` (use the transitive upper bound). This is the safe direction — it may overcount, causing the gate to fail when it should pass, which triggers human review.

2. **Parse failure:** If the test source file is found but the method body cannot be parsed (e.g., the regex approach fails on complex Kotlin syntax), `M_p(t)` is **indeterminable**. The conservative fallback is `M_p(t) = M_t(t)`.

3. **Classification ambiguity:** If a callee cannot be classified as production or non-production (e.g., the method name exists in both production and test sources, or the callee's defining class cannot be resolved), the callee is classified as a **production method** (conservative — may overcount `M_p`, which is the safe direction for a granularity gate).

4. **Reporting:** When the fallback is used, the `check-tier.sh` script MUST emit a WARNING: `"M_p for test <test_name> is indeterminable (source analysis failure); using transitive upper bound M_t = <value>"`. The tier report should include a count of tests for which source analysis failed.

### 2.5 M_t — Transitive Methods Per Test (Informational Diagnostic; formerly Gate 3b)

> **Note:** In v3.0, M_t is NO LONGER a gate. It is reported as an informational diagnostic. See Axiom 3 for the rationale.

#### 2.5.1 Definition

**M_t(t)** for a test `t` is the number of **distinct production methods** whose mutations are killed by `t`. This is the transitive closure of the test's reach through the production call graph — it counts all methods whose mutations the test kills, whether directly or transitively.

Formally:

```
M_t(t) = |{ (c, m) : exists a killed mutation mu such that mu.mutatedClass = c and mu.mutatedMethod = m and t in killingTests(mu) }|
```

The metric **M_t** for the test suite is:

```
M_t = max_{t in T} M_t(t)
```

**Note:** In v3.0, M_t is NOT a gate. This value is reported as an informational diagnostic only. It does not affect tier calculation. The thresholds below are retained for reference but are not enforced.

#### 2.5.2 Data Source

**PIT XML report** with `fullMutationMatrix=true`.

**XML path:** For each `<mutation>` element with `status="KILLED"`, the `killingTests` child element contains a pipe-separated (`|`) list of test method names that kill this mutation. The `mutatedClass` and `mutatedMethod` child elements identify the method being mutated.

#### 2.5.3 Computation

1. Parse the PIT XML with `fullMutationMatrix=true`.
2. For each test `t` (identified from `killingTests` entries across all mutations), initialize `methods_killed_by_t = {}` (empty set).
3. For each `<mutation>` element with `status="KILLED"`:
   a. Read `mutatedClass` -> `c`.
   b. Read `mutatedMethod` -> `m`.
   c. Read `killingTests` -> split by `|` -> list of test method names.
   d. For each test `t` in the list:
      - Add `(c, m)` to `methods_killed_by_t[t]`.
4. For each test `t`: `M_t(t) = |methods_killed_by_t[t]|`.
5. `M_t = max_t M_t(t)`.
6. If no mutations are killed (all tests have `M_t(t) = 0`), `M_t = 0`.

**Unit:** Integer (count of distinct methods).

**PIT format:** Each entry in the `killingTests` field is formatted as `[method:methodName()]`. The test method name is extracted by stripping the `[method:` prefix and `()]` suffix.

**Legacy format:** If `fullMutationMatrix=false`, PIT emits a `killingTest` (singular) element containing only the first test that killed the mutation. In this case, `M_t` is an **undercount** (some tests that kill mutations are not reported). The system MUST require `fullMutationMatrix=true` for R_direct (Gate 3b) to be evaluated accurately. Without it, R_direct is computed from the available data but flagged as unreliable, and the system falls back to R with lenient thresholds.

#### 2.5.4 What M_t Counts (Synthetic and Lambda Methods)

`M_t` counts **all** distinct `(mutatedClass, mutatedMethod)` pairs from killed mutations, including:

- **User-defined methods:** Standard production methods written by developers.
- **Synthetic methods:** Compiler-generated methods (e.g., Kotlin data class `copy$default`, `access$super`, companion object methods). These are counted because they represent real bytecode that the test exercises.
- **Lambda bodies:** Kotlin lambdas compile to `invoke` or `invoke$default` methods in synthetic classes (e.g., `Foo$lambda$1`). These are counted as distinct methods.
- **Inline function bodies:** Kotlin inline functions may generate standalone method bodies. These are counted.

**Rationale for counting synthetic/lambda methods in M_t:** `M_t` measures the transitive *reach* of a test through the production call graph. If a test calls a method that invokes 10 lambdas, each of which has mutations killed by the test, the test's transitive reach is 10+ methods. This is real coarseness — the test is exercising a large surface area, even if some of it is compiler-generated. The test should be broken into smaller tests or the production code should be decomposed.

**Exception — data class generated methods:** Kotlin data class generated methods (`componentN`, `copy`, `toString`, `hashCode`, `equals`) are often excluded from PIT via configuration because they are equivalent mutants or trivially correct. If they are excluded from PIT, they generate no mutations and are invisible to `M_t`. If they are NOT excluded, they are counted. The project should document in `TEST_STRATEGY.md` whether data class methods are excluded from PIT and why.

**Deduplication:** A method is identified by the `(mutatedClass, mutatedMethod)` pair. If a single method has multiple killed mutations (e.g., 5 mutations in `Cart.checkout()`, all killed by `testCheckout()`), it counts as **1** method toward `M_t(testCheckout)`, not 5. `M_t` counts distinct methods, not distinct mutations.

#### 2.5.5 M_t and Mega-Test Blocking

`M_t` is reported as an **informational diagnostic** in v3.0. While it is no longer a gate, it provides useful signal about test integration depth:

- A high `M_t` with low `M_p` indicates the test exercises one entry point that reaches many methods (orchestrator integration test). This is expected for Gradle plugins and other orchestrator-heavy architectures.
- A high `M_t` with high `M_p` indicates the test calls many methods directly AND reaches many more transitively. This is caught by M_p alone.
- `M_p` (Gate 3a) is the sole granularity gate in v3.0. It catches tests that directly call too many production methods.

### 2.6 R — Killers Per Mutation (Fallback for Gate 3b: Redundancy)

> **Note:** In v3.0, R is replaced by R_direct (Section 2.6.1) when source analysis is available. R is used as a fallback with lenient thresholds when source analysis is not available.

#### 2.6.1 Definition

**R(m)** for a mutation `m` is the number of **distinct tests** that kill `m`.

A test `t` **kills** mutation `m` if, when `m` is applied to the production code, test `t` fails (assertion error, exception, or non-zero exit).

The metric **R** for the test suite is:

```
R = max_{m in M_killed} R(m)
```

where `M_killed` is the set of killed mutations. (Survived or no-coverage mutations have `R(m) = 0` by definition and are excluded from the max.)

The fallback gate checks: `R <= R_direct_threshold (or R fallback threshold)` where `R_direct_threshold (or R fallback threshold)` is the tier threshold (lenient values to accommodate transitive kill inflation). When source analysis is available, R_direct (Section 2.6.1) is used instead with tighter thresholds.

#### 2.6.2 Data Source

**PIT XML report** with `fullMutationMatrix=true`.

**XML path:** For each `<mutation>` element with `status="KILLED"`, the `killingTests` child element contains a pipe-separated (`|`) list of test method names that kill this mutation.

**PIT format:** Each entry in the `killingTests` field is formatted as `[method:methodName()]`. The test method name is extracted by stripping the `[method:` prefix and `()]` suffix.

**Legacy format:** If `fullMutationMatrix=false`, PIT emits a `killingTest` (singular) element containing only the first test that killed the mutation. In this case, `R(m)` can only be determined to be `>= 1`, and the true `R` is unknown. The system MUST require `fullMutationMatrix=true` for Gate 3b to be evaluated accurately. Without it, the system uses R = 1 (trivially passing all thresholds).

#### 2.6.3 Computation

1. Parse the PIT XML with `fullMutationMatrix=true`.
2. For each `<mutation>` element with `status="KILLED"`:
   a. Read the `killingTests` child element.
   b. If `killingTests` is absent or empty, `R(m) = 0` (treat as data error — a KILLED mutation should have at least one killing test).
   c. Split the text content by `|` to get individual test entries.
   d. For each entry, extract the test method name (strip `[method:` prefix and `()]` suffix).
   e. Deduplicate the test method names (PIT should not report duplicates, but deduplication is defensive).
   f. `R(m) = |{distinct test method names}|`.
3. Compute `R = max_{m in M_killed} R(m)`.
4. If `M_killed` is empty (no mutations killed), `R = 0`.

**Unit:** Integer (count of tests).

#### 2.6.4 R and Transitive Kills

When production method `A.foo()` calls `B.bar()`, a test for `A.foo()` will transitively execute `B.bar()`. If `B.bar()` has mutations, and the test for `A.foo()` kills them, then `R(m) >= 2` for `B.bar()`'s mutations (killed by both `testA` and `testB`). This inflates `R` for `B.bar()`'s mutations.

**Canonical handling:** Transitive kills are counted in `R` as-is. There is no "transitive kill discount." The rationale is that if `testA` kills `B.bar`'s mutations, it is redundant with `testB` for those mutations, and the test suite has redundancy. The fix is to either (a) mock `B.bar` in `testA` so it doesn't kill `B.bar`'s mutations, or (b) accept the redundancy and ensure `R` is within the threshold. This is a design constraint on the test suite, not a computation artifact.

**R_direct at Perfection (<=3):** The Perfection tier requires `R_direct <= 3`. The rationale:

- `R_direct` counts only tests that DIRECTLY call the mutated method. Transitive kills (from orchestrator tests) are NOT counted.
- `R_direct <= 3` allows up to 3 tests to directly call and kill the same mutation, accommodating: 1 primary test + 1-2 secondary tests that also directly call the method for different scenarios.
- This is the practical minimum for non-trivial codebases.
- The old `R` (all killers) required `<= 5` at Perfection in fallback mode, because transitive kills from orchestrator tests inflated the count to 3-5 for shared utility methods. R_direct eliminates this inflation.

#### 2.6.5 R_direct — Direct Killers Per Mutation (Preferred Gate 3b Metric)

**Definition:** `R_direct(m)` = the number of tests that BOTH kill mutation `m` AND directly call the method where `m` resides (the mutated method is in the test's primary method set).

**Computation:**

1. For each killed mutation `m` in method `(cls, method)`:
   a. Get the set of killing tests: `killers(m) = {tests that kill m}` (from PIT's `killingTests`).
   b. For each test `t` in `killers(m)`, check if `method` is in `t`'s primary method set `P(t)` (from source analysis, Section 2.4).
   c. `R_direct(m) = |{t in killers(m) : method in P(t)}|`.
2. Compute `R_direct = max_{m in M_killed} R_direct(m)`.

**Key difference from R:** R counts ALL killing tests (including transitive). R_direct counts only killing tests that directly call the mutated method. This eliminates structural inflation from transitive kills:

- If `testApply()` calls `apply()` which calls `filterExistingDirs()`, and `testApply()` kills a mutation in `filterExistingDirs()`, that kill is TRANSITIVE.
- `testApply()` does NOT have `filterExistingDirs` in its primary method set (it has `apply` instead).
- So `testApply()` is NOT counted in `R_direct` for `filterExistingDirs()` mutations.
- Only `testFilterExistingDirs()` (which directly calls `filterExistingDirs`) is counted.

**Availability:** R_direct requires source analysis (Phase 2, `--test-src` flag). Without source analysis, the system falls back to R with lenient thresholds (Section 4.1).

**Fallback R thresholds (v3.0):** When source analysis is not available, R is used with these lenient thresholds that accommodate structural transitive kill inflation:

| Tier | R_direct (preferred) | R (fallback) |
|------|---------------------|-------------|
| Bronze | <= 10 | <= 20 |
| Silver | <= 6 | <= 12 |
| Gold | <= 4 | <= 8 |
| Platinum | <= 3 | <= 6 |
| Perfection | <= 3 | <= 5 |

### 2.7 Test Speed (Unit)

**Formula:**

```
unit_test_speed = total_unit_test_time_ms / unit_test_count
```

**Data source:** JUnit XML reports from the unit test task (e.g., `build/test-results/test/*.xml`).

**XML path:** Each `<testsuite>` element has `tests` (attribute: total test count) and `time` (attribute: wall-clock time in seconds) attributes.

**Computation:**

1. Glob all JUnit XML files in the unit test results directory.
2. For each `<testsuite>` element:
   a. `unit_test_count += int(tests attribute)`.
   b. `total_unit_test_time_seconds += float(time attribute)`.
3. `total_unit_test_time_ms = total_unit_test_time_seconds * 1000`.
4. `unit_test_speed = total_unit_test_time_ms / unit_test_count` if `unit_test_count > 0`, else `N/A`.
5. If `unit_test_count == 0`, the metric is `N/A` (no unit tests). The metric is not evaluated and does not block any tier.

**Unit:** Milliseconds per test, one decimal place.

**Semantics:** Wall-clock time per test, averaged across all unit tests. Excludes compile time, PIT time, and any non-test execution time. The JUnit `time` attribute for a `<testsuite>` is the total time for all test cases in that suite, so summing across suites and dividing by total test count gives the per-test average.

### 2.8 Test Speed (Integration)

**Formula:**

```
integration_test_speed = total_integration_test_time_ms / integration_test_count
```

**Data source:** JUnit XML reports from the integration test task (e.g., `build/test-results/integrationTest/*.xml` or a directory specified via `--integration-junit-dir`).

**Computation:** Identical to unit test speed (Section 2.7), but using the integration test JUnit XML directory.

**Unit:** Milliseconds per test, one decimal place.

**N/A handling:** If the project has no integration tests (`integration_test_count == 0`), the metric is `N/A` and does not block any tier. A project without integration tests is not penalized.

### 2.9 Exclusion Ratio

**Formula:**

```
exclusion_ratio = excluded_instructions / (excluded_instructions + included_instructions) * 100
```

**Data source:** JaCoCo XML report (for `included_instructions`) and compiled `.class` files on disk (for `excluded_instructions`).

**Computation:**

1. `included_instructions = instruction_covered + instruction_missed` (from JaCoCo root-level `<counter type="INSTRUCTION">`).
2. Identify the compiled class output directory (e.g., `build/classes/kotlin/main`, `build/classes/java/main`).
3. Walk the class directory tree. For each `.class` file:
   a. Normalize the class name (strip `.class` suffix, convert path separators to `/`).
   b. Check if the class (or its outer class for inner classes — strip `$` suffix) is present in the JaCoCo XML `<class>` elements.
   c. If present, the class is **included** — skip it.
   d. If not present, the class is **excluded** — count its bytecode instructions using `javap -c` (count lines that look like bytecode instructions: non-empty, contain `:`, and are not comments).
   e. `excluded_instructions += instruction_count`.
4. `exclusion_ratio = excluded_instructions / (excluded_instructions + included_instructions) * 100`.

**Unit:** Percentage, one decimal place.

**Semantics:** Measures the fraction of production bytecode that is excluded from coverage measurement. A high exclusion ratio indicates that the project is excluding significant code from coverage, which may be gaming the coverage metrics. Exclusions are legitimate for equivalent mutants, generated code, and code that cannot be meaningfully tested (e.g., DI module classes), but the ratio tracks how much is excluded.

**Inner class handling:** Inner classes (e.g., `Foo$Bar`) are attributed to their outer class (`Foo`). If the outer class is in JaCoCo, the inner class is considered included. This prevents Kotlin companion objects, data class generated methods, and other compiler-generated inner classes from inflating the exclusion ratio.

**Implementation requirement:** The exclusion ratio gate MUST be implemented in `check-tier.sh` and MUST be evaluated at all tiers. The implementation must:
- Walk the compiled class output directory (configurable via `--class-dir` argument, default: `build/classes/kotlin/main` with fallback to `build/classes/java/main`).
- For each `.class` file not in the JaCoCo XML, count bytecode instructions using `javap -c`.
- Inner classes (`$` in name) are attributed to their outer class.
- The exclusion ratio is computed and compared against the tier threshold.
- If the exclusion ratio exceeds the threshold, the tier gate fails.

### 2.10 Test Isolation

**Definition:** Tests pass when run 3+ times with randomized execution order. No test depends on shared mutable state or execution order.

**Verification:** Automated. The CI pipeline runs the test suite 3 times with `--tests-shuffle` (or equivalent) and verifies all runs pass. Alternatively, the presence of a `TEST_ISOLATION.md` or `TEST_STRATEGY.md` file that documents the isolation strategy is accepted as evidence at Bronze and Silver tiers (where test isolation is optional). At Gold and above, automated verification is required.

**Data source:** CI pipeline configuration and `TEST_ISOLATION.md` / `TEST_STRATEGY.md` file presence.

**Computation:** Boolean (`true` / `false`).

### 2.11 Documented Strategy

**Definition:** A `TEST_STRATEGY.md` file is committed to the repository root. The document is a decision document describing what is tested, how, and why — not a tutorial.

**Verification:** File existence check.

**Data source:** Filesystem.

**Computation:** Boolean (`true` / `false`).

## 3. Edge Case Catalog

This section enumerates every known edge case and defines the canonical handling. Every ambiguity is a bug in the spec; this catalog exists to eliminate ambiguity.

### 3.1 Zero-Kill Test (Test That Kills 0 Mutations)

**Definition:** A test `t` such that no mutation in the PIT XML has `t` in its `killingTests` list.

**Effect on M_p (Gate 3a):**
- `M_t(t) = 0` (no methods' mutations killed).
- `M_p(t) = 0` (no primary methods identified via PIT).
- A zero-kill test does not contribute to `M_p` (the max is taken over all tests; a test with `M_p(t) = 0` cannot be the maximum unless all tests are zero-kill, in which case `M_p = 0 <= M_p_threshold` and the gate passes trivially — but Axiom 2 would fail because mutation score = 0%).

**Effect on M_t (informational):**
- `M_t(t) = 0` (no methods' mutations killed).
- A zero-kill test does not contribute to `M_t`.

**Effect on R (Gate 3b (fallback)):**
- A zero-kill test does not appear in any mutation's `killingTests` list, so it does not contribute to `R(m)` for any `m`.
- Zero-kill tests are invisible to Gate 3b (fallback).

**Effect on Axiom 2:**
- Zero-kill tests do not help satisfy Axiom 2. They are pure waste.
- A zero-kill test that exercises covered instructions (Axiom 1) is a "coverage-only test" — it contributes to coverage but not to mutation killing. See Section 3.9.

**Canonical handling:** Zero-kill tests are not directly penalized by Gates 3a or 3b. They are penalized indirectly: they increase `unit_test_count` (lowering test speed) and consume CI time without contributing to mutation score. A test suite with many zero-kill tests will struggle to achieve high mutation score (Axiom 2) and will have poor test speed.

### 3.2 Method With 0 Mutations

**Definition:** A production method that appears in the JaCoCo XML (has instructions and/or branches) but has no mutations in the PIT XML.

**Causes:**
1. The method contains no mutatable bytecode patterns (e.g., the method body is a single `return` or a single field assignment that PIT's mutators do not target).
2. The method's class is in PIT's `avoidCallsTo` list (e.g., logging methods, `toString` methods).
3. The method's class is excluded from PIT configuration.
4. The method is synthetic/compiler-generated (e.g., Kotlin data class `copy()`, `componentN()`).

**Effect on M_p (Gate 3a):**
- If a test calls a method with 0 mutations, that method will not appear in the PIT kill matrix (no mutations to kill). The PIT fast-path (Phase 1) will not count it.
- The source analysis (Phase 2) WILL count it as a primary method (it is a direct call to a production method).
- This means `M_p(t)` can be higher than `M_t(t)` for tests that call unmutatable methods. This is the correct behavior — the test is still exercising a production method directly, and granularity should account for it.

**Effect on M_t (informational):**
- A method with 0 mutations contributes no mutations to the kill matrix. It is invisible to `M_t`.
- This means `M_t` may undercount the true transitive reach of a test if some transitively-reached methods have no mutations. This is a known limitation (Section 8.1).

**Effect on R (Gate 3b (fallback)):**
- A method with 0 mutations contributes no mutations to the `R` computation. It is invisible to Gate 3b (fallback).

**Effect on Axiom 2:**
- A method with 0 mutations does not contribute to `total_mutations` or `killed_mutations`. It is invisible to Axiom 2.

**Effect on Axiom 1:**
- A method with 0 mutations still has instructions and branches that must be covered. It is fully in scope for Axiom 1.

**Warning in CI:** The `check-tier.sh` script emits a WARNING for classes that have JaCoCo branches but no PIT mutations. This warning is informational and does not fail the build. The project should document in `TEST_STRATEGY.md` why these classes have no mutations (exclusion, `avoidCallsTo`, or no mutatable patterns).

### 3.3 Transitive Kill Coverage (Test for A Kills B's Mutations)

**Definition:** Test `testA()` is written to test method `A.foo()`. `A.foo()` calls `B.bar()`. When `testA()` runs, it kills mutations in both `A.foo()` and `B.bar()` (because `B.bar()` is executed transitively).

**Effect on M_p (Gate 3a):**
- PIT fast-path: `M_t(testA) = 2` (mutations in both `A.foo` and `B.bar` were killed).
- Source analysis: `M_p(testA) = 1` (only `A.foo` is directly called; `B.bar` is transitive).
- If `M_p_threshold = 1`, the PIT fast-path would flag `testA` as exceeding the threshold (`2 > 1`), but source analysis corrects it to `1 <= 1` -> PASS.
- This is the primary reason the hybrid computation exists.

**Effect on M_t (informational):**
- `M_t(testA) = 2` (both `A.foo` and `B.bar` mutations were killed by `testA`).
- In v3.0, M_t is informational only and does NOT block any tier. The test reaches 2 methods transitively, which is expected for integration testing.
- The fix is to mock `B.bar()` in `testA` so that `testA` does not kill `B.bar`'s mutations, reducing `M_t(testA)` to 1.

**Effect on R (Gate 3b (fallback)):**
- Mutations in `B.bar` that are killed by `testA` will have `testA` in their `killingTests` list.
- If `testB()` (a test for `B.bar`) also kills the same mutations, then `R(m) >= 2` for those mutations.
- `testA` is a transitive killer of `B.bar`'s mutations. This inflates `R` for `B.bar`'s mutations.
- This is a known limitation (Section 8.2): `R = 1` may be unreachable when methods call each other, because the test for the caller transitively kills the callee's mutations. The Perfection tier accommodates this with `R <= 3` (not `R <= 1`).

**Canonical handling:** Transitive kills are counted in `R` as-is. There is no "transitive kill discount." The rationale is that if `testA` kills `B.bar`'s mutations, it is redundant with `testB` for those mutations, and the test suite has redundancy. The fix is to either (a) mock `B.bar` in `testA` so it doesn't kill `B.bar`'s mutations, or (b) accept the redundancy and ensure `R` is within the threshold. This is a design constraint on the test suite, not a computation artifact.

### 3.4 EqualKillSet Groups (Multiple Tests With Identical Kill Sets)

**Definition:** Two or more tests `t1, t2, ..., tk` such that `killingSet(t1) = killingSet(t2) = ... = killingSet(tk)` (they kill exactly the same set of mutations).

**Effect on M_p (Gate 3a):**
- Each test in the group is evaluated independently. If each test has `M_p(t) <= M_p_threshold`, the gate passes. Equal-kill-set tests are not inherently coarse.

**Effect on M_t (informational):**
- Each test in the group has the same `M_t(t)` (same set of methods' mutations killed). If `M_t(t) <= M_t_threshold`, the gate passes.

**Effect on R (Gate 3b (fallback)):**
- For every mutation `m` in the shared kill set, `R(m) >= k` (all `k` tests kill it).
- If `k > R_direct_threshold (or R fallback threshold)`, the gate fails for those mutations.
- Equal-kill-set groups are the most common cause of Gate 3b (fallback) failures.

**Canonical handling:** Equal-kill-set groups are treated as redundancy. If `k` tests kill the same mutations, `k - 1` of them are redundant (assuming any single test is sufficient to kill the mutations). The gate fails if `k > R_direct_threshold (or R fallback threshold)`. The fix is to remove `k - R_direct_threshold (or R fallback threshold)` tests from the group, or to differentiate the tests so they kill different mutations.

**Special case — identical kill set of size 0:** If `k` tests all kill 0 mutations, they form a zero-kill group. See Section 3.1. They do not affect `R` (they are not in any mutation's `killingTests`).

### 3.5 Lambda and Synthetic Methods in Kotlin

**Definition:** Kotlin generates synthetic methods for lambdas, inline functions, data class generated methods (`copy`, `componentN`, `toString`, `hashCode`, `equals`), companion object methods, and coroutines. These appear in JaCoCo and PIT as methods with names like `invoke`, `copy$default`, `access$super`, etc.

**Effect on M_p (Gate 3a):**
- Synthetic methods are production methods if they are in production classes. A test that calls a method that in turn invokes a lambda will have the lambda's mutations in its kill set (transitive).
- In source analysis (Phase 2), synthetic methods are NOT counted as primary methods because they are not directly called from the test source — they are invoked by the Kotlin runtime or by other production methods.
- Exception: if the test directly invokes a lambda (e.g., `val f: (Int) -> Int = { it + 1 }; f(42)`), the lambda's `invoke` method is a primary method. However, this is rare and the source analysis should count it.

**Effect on M_t (informational):**
- Synthetic methods' mutations ARE counted in `M_t`. If a test kills mutations in 5 user-defined methods and 3 lambda `invoke` methods, `M_t(t) = 8`.
- This is the correct behavior: the test is transitively reaching 8 distinct production methods, including compiler-generated ones. The test's transitive footprint is 8.
- **Exception:** If synthetic methods are excluded from PIT (via `excludedClasses` or `excludedMethods` configuration), they generate no mutations and are invisible to `M_t`. The project should document such exclusions in `TEST_STRATEGY.md`. The exclusion ratio (Section 2.9) tracks the impact.

**Effect on R (Gate 3b (fallback)):**
- Synthetic methods' mutations are counted in `R` like any other mutation. If a lambda's mutation is killed by multiple tests (because the lambda is called from multiple paths), `R` for that mutation may be high.

**Effect on Axiom 1:**
- Synthetic methods have instructions and branches that must be covered. They are fully in scope.

**Effect on Axiom 2:**
- Synthetic methods' mutations must be killed. They are fully in scope.

**Canonical handling:** Synthetic methods are treated identically to user-defined methods for Axioms 1 and 2. For Gate 3a (M_p), they are excluded from primary method count in source analysis (they are not directly called from test source). For the M_t informational diagnostic, they are counted normally (they contribute to the transitive footprint). For Gate 3b (R_direct), they are counted normally. Projects may exclude synthetic methods from PIT if they are equivalent mutants or untestable, but this increases the exclusion ratio (Section 2.9).

**Identification:** Synthetic methods in Kotlin are identified by:
- Method names containing `$` (e.g., `copy$default`, `access$super`).
- Method names matching `invoke`, `invoke$default` (lambdas).
- Method names matching `component1`, `component2`, etc. (data class destructuring).
- Methods in classes ending in `$Companion`, `$Lambda`, `$serializer`, etc.
- The `ACC_SYNTHETIC` flag in the bytecode (can be checked via `javap -v`).

**Data class getters — formal rule:** Kotlin data class generated methods (`componentN`, `copy`, `toString`, `hashCode`, `equals`) are classified as follows:
- **M_p (Gate 3a):** NOT counted as primary methods in source analysis. A test that constructs a data class instance and accesses its properties (e.g., `val p = Point(1, 2); assertEquals(1, p.x)`) does NOT have `component1` or `getX` as a primary method. The constructor is the primary method (if it is a production method).
- **M_t (informational):** Counted if they have mutations in PIT. If the data class is in PIT's scope and the generated methods have mutatable bytecode, their mutations are counted in `M_t`. If the data class is excluded from PIT, they are invisible. Does NOT affect tier calculation.
- **R (Gate 3b (fallback)):** Counted if they have mutations in PIT.
- **Axiom 1:** Fully in scope (instructions and branches must be covered).
- **Axiom 2:** Fully in scope (mutations must be killed) unless excluded from PIT.

### 3.6 Integration Tests vs Unit Tests

**Definition:**
- **Unit test:** A test that exercises a single unit of code (class or method) in isolation, with all external dependencies mocked or stubbed. Runs in the JVM (no emulator, no device, no network). Fast (< 200ms per test at Bronze).
- **Integration test:** A test that exercises multiple units together, or that interacts with external systems (database, network, filesystem, emulator). Slower than unit tests.

**Separation:** The project MUST separate unit and integration tests into different source sets and Gradle tasks (e.g., `test` for unit tests, `integrationTest` for integration tests). The JUnit XML reports MUST be in separate directories.

**Effect on metrics:**
- **Test speed (Section 2.7, 2.8):** Unit test speed and integration test speed are computed separately and have separate thresholds. A slow integration test does not penalize unit test speed and vice versa.
- **M_p (Gate 3a):** Applied to ALL tests (unit and integration). Integration tests typically have higher `M_p` because they exercise multiple components directly. The threshold `M_p_threshold` is the same for both — if integration tests exceed `M_p_threshold`, they must be broken into smaller tests or the threshold must be relaxed (which lowers the tier).
- **M_t (informational):** Applied to ALL tests (unit and integration). Integration tests typically have much higher `M_t` because they exercise many components transitively. This is expected and NO LONGER BLOCKS ANY TIER in v3.0. M_t is reported as an informational diagnostic only.
- **R_direct (Gate 3b):** Applied to ALL tests (unit and integration). Only tests that DIRECTLY call the mutated method are counted. If a unit test and an integration test both directly call and kill the same mutation, `R_direct(m) >= 2`. Transitive kills from integration tests (where the test calls a different method that calls the mutated method) are NOT counted.
- **Mutation score (Axiom 2):** PIT runs against the full test suite (unit + integration). All tests contribute to killing mutations.
- **Coverage (Axiom 1):** JaCoCo aggregates coverage from all test runs. Both unit and integration tests contribute to coverage.

**N/A handling:** If a project has no integration tests, the integration test speed metric is `N/A` and does not block any tier. The project is not penalized for not having integration tests.

### 3.7 Tests for Code in PIT's `avoidCallsTo` List

**Definition:** PIT's `avoidCallsTo` configuration directive tells PIT to not generate mutations for methods that call certain classes (e.g., logging frameworks, `System.out`, `println`). This prevents PIT from generating mutations that are untestable (e.g., removing a log statement should not cause a test failure).

**Effect on M_p (Gate 3a):**
- Methods in `avoidCallsTo` classes have no mutations. Tests that call these methods will not see them in the PIT kill matrix.
- Source analysis (Phase 2) will count these as primary methods if they are directly called from the test. This is correct — the test is exercising the method, even if PIT can't mutate it.

**Effect on M_t (informational):**
- No mutations are generated for `avoidCallsTo` methods, so they do not contribute to `M_t`.
- `M_t` may undercount the true transitive reach of a test if some transitively-reached methods are in `avoidCallsTo`. This is a known limitation.

**Effect on R (Gate 3b (fallback)):**
- No mutations are generated for `avoidCallsTo` methods, so they do not contribute to `R`.

**Effect on Axiom 2:**
- No mutations are generated, so these methods are invisible to Axiom 2.

**Effect on Axiom 1:**
- These methods still have instructions and branches that must be covered. They are fully in scope for Axiom 1.

**Canonical handling:** `avoidCallsTo` methods are treated like any method with 0 mutations (Section 3.2). The project should document in `TEST_STRATEGY.md` which classes are in `avoidCallsTo` and why. The `check-tier.sh` script's warning for "classes with branches but no mutations" covers this case.

### 3.8 Partial PIT Runs (`partial="true"`)

**Definition:** PIT can run in partial mode, where it does not generate or test all mutations. The PIT XML root element may have a `partial="true"` attribute, indicating that the mutation results are incomplete.

**Causes:**
1. PIT was interrupted (timeout, CI kill, OOM).
2. PIT was run with `--maxMutationsPerClass` or similar limiting flags.
3. PIT was run with `--historyInput` / `--historyOutput` for incremental mutation testing.

**Effect on all metrics:**
- `total_mutations` and `killed_mutations` are incomplete. The mutation score (Axiom 2) is unreliable.
- The kill matrix is incomplete. `M_p`, `M_t`, and `R` are unreliable.
- Coverage (Axiom 1) is unaffected (JaCoCo runs independently of PIT).

**Canonical handling:** The `check-tier.sh` script MUST detect `partial="true"` on the PIT XML root element and:
1. Emit a WARNING: "PIT run is partial — mutation metrics are unreliable."
2. Evaluate Axiom 1 (coverage) and test speed normally (these are unaffected).
3. Evaluate Axiom 2 (mutation score) and Axiom 3 (M_p, M_t, R) with a flag indicating the data is partial.
4. If `partial="true"`: the tier is capped at **Silver** regardless of other metrics. A partial run cannot certify Gold or above because the mutation metrics are incomplete. The build does not fail (exit 0) but the tier report shows the cap.
5. The project MUST re-run PIT with a full mutation matrix (no partial) to certify Gold or above.

**Detection:** Parse the PIT XML root element. If the `partial` attribute exists and its value is `"true"`, the run is partial. If the attribute is absent or `"false"`, the run is full.

### 3.9 Coverage-Only Tests (Tests that Exercise Code But Kill No Mutations)

**Definition:** A test that executes production code (contributing to instruction and branch coverage) but does not kill any mutations. This happens when the test exercises the code path but does not assert on the behavior — the code runs, but if the code were mutated, the test would still pass.

**Example:** A test that calls `service.process(data)` but does not assert on the return value or any side effect. The instructions in `service.process` are covered, but a mutation (e.g., changing `+` to `-` in the calculation) would not cause the test to fail.

**Effect on M_p (Gate 3a):**
- `M_t(t) = 0` (no mutations killed).
- `M_p(t) > 0` (the test does call production methods, but since no mutations are killed, the PIT fast-path returns 0).
- **Discrepancy:** The PIT fast-path undercounts `M_p` for coverage-only tests because it relies on killed mutations. Source analysis (Phase 2) would count the primary methods, but Phase 2 is only triggered when `M_t(t) > M_p_threshold`. Since `M_t(t) = 0 <= M_p_threshold`, Phase 2 is never triggered.
- **Result:** Coverage-only tests are invisible to Gate 3a. They do not inflate `M_p`.

**Effect on M_t (informational):**
- `M_t(t) = 0` (no mutations killed). Coverage-only tests are invisible to Gate 3b.

**Effect on R (Gate 3b (fallback)):**
- Coverage-only tests kill 0 mutations, so they do not appear in any `killingTests` list. They are invisible to Gate 3b (fallback).

**Effect on Axiom 1:**
- Coverage-only tests contribute to instruction and branch coverage. They help satisfy Axiom 1.

**Effect on Axiom 2:**
- Coverage-only tests do not kill mutations. They do not help satisfy Axiom 2.

**Canonical handling:** Coverage-only tests are invisible to all three Axiom 3 gates (M_p, M_t, and R). They are penalized indirectly: they consume CI time (lowering test speed) and do not contribute to mutation score. A test suite with many coverage-only tests will have high coverage but low mutation score, failing Axiom 2. This is a known limitation (Section 8.3): the Axiom 3 gates are mutation-based and cannot detect tests that exercise code without asserting on it. The mutation score (Axiom 2) is the primary defense against coverage-only tests.

**Diagnostic:** The `check-tier.sh` script should emit a diagnostic when `instruction_coverage - mutation_score > 25%` (coverage is much higher than mutation score, suggesting coverage-only tests). This diagnostic is informational, not a gate failure.

### 3.10 Facade Mega-Test (M_p Passes, M_t Fails)

**Definition:** A test that directly calls a small number of production methods (passing Gate 3a) but those methods are facades that fan out to many production methods transitively (failing Gate 3b).

**Example:**
- Test `testWorkflow()` calls `WorkflowRunner.run(config)`.
- `WorkflowRunner.run()` internally calls `Parser.parse()`, `Validator.validate()`, `Executor.execute()`, `Logger.log()`, `Reporter.report()`, `Cache.store()`, `Notifier.notify()`.
- `M_p(testWorkflow) = 1` (only `WorkflowRunner.run()` is directly called).
- `M_t(testWorkflow) = 8` (mutations in all 8 methods are killed by `testWorkflow`).
- At Gold tier: `M_p_threshold = 3`. Gate 3a passes (`1 <= 3`). M_t = 8 is informational only and does NOT block the tier in v3.0.

**Canonical handling (v3.0):** In v2.0, this test would have failed Gate 3b (M_t). In v3.0, M_t is no longer a gate. This test is a **legitimate integration test** for the facade entry point. It passes all gates in v3.0. If the developer wants to verify each internal method independently, they should add separate focused tests for each — but the integration test is not penalized.

**Note:** This test IS still caught if it directly calls too many methods (M_p > threshold). In this example, M_p = 1, so it passes. The facade pattern is acceptable in v3.0 because M_t has been removed as a gate.

### 3.11 R Inflated by Transitive Kills

**Definition:** `R` for a mutation `m` in method `B.bar()` is inflated because `testA` (a test for `A.foo()`, which calls `B.bar()`) transitively kills `m`, in addition to `testB` (the direct test for `B.bar()`).

**Example:**
- `A.foo()` calls `B.bar()`.
- `testA` tests `A.foo()` and transitively kills mutations in `B.bar()`.
- `testB` tests `B.bar()` directly.
- `testIntegration` is an integration test that exercises both `A.foo()` and `B.bar()`.
- For a mutation `m` in `B.bar()`: `R(m) = 3` (killed by `testA`, `testB`, and `testIntegration`).

**Effect on Gate 3b (fallback):**
- If `R_direct_threshold (or R fallback threshold) = 3` (Perfection), `R(m) = 3 <= 3` -> PASS.
- If `R_direct_threshold (or R fallback threshold) = 2` (would be required for a stricter tier), `R(m) = 3 > 2` -> FAIL.
- The Perfection tier sets `R_direct_threshold (or R fallback threshold) = 3` specifically to accommodate this pattern: 1 direct test + 1 transitive killer + 1 integration test = 3 killers.

**Canonical handling (v3.0):** With R_direct (preferred Gate 3b metric), transitive kills are NOT counted. `testA` does not directly call `B.bar()`, so it is excluded from `R_direct(m)`. Only `testB` (which directly calls `B.bar()`) is counted: `R_direct(m) = 1`. The transitive kill inflation problem is eliminated.

When source analysis is unavailable (R fallback), transitive kills ARE counted in R. The lenient fallback thresholds (Section 4.1) accommodate this inflation. Options for reducing R in fallback mode:
1. Mock `B.bar()` in `testA` so `testA` does not kill `B.bar()`'s mutations.
2. Accept `R(m) = 3` and ensure the fallback tier threshold is `>= 3` (all tiers in v3.0 have fallback R >= 5).

## 4. Tier Thresholds

A repository's tier is the **highest** tier where **ALL** required metrics are met. If any metric for a tier is not met, the repository cannot be at that tier or any tier above it.

### 4.1 Tier Table

| Metric | Bronze | Silver | Gold | Platinum | Perfection |
|--------|--------|--------|------|----------|-------------|
| Instruction coverage | >= 60% | >= 80% | >= 90% | >= 95% | **100%** |
| Branch coverage | >= 50% | >= 70% | >= 85% | >= 90% | **100%** |
| Mutation score | >= 50% | >= 70% | >= 85% | >= 95% | **100%** |
| M_p (max primary methods per test) | <= 5 | <= 4 | <= 3 | <= 2 | **<= 2** |
| R_direct (max direct killers per mutation) | <= 10 | <= 6 | <= 4 | <= 3 | **<= 3** |
| M_t (max transitive methods per test) | info | info | info | info | **info** |
| Unit test speed | < 200ms | < 100ms | < 50ms | < 30ms | **< 10ms** |
| Integration test speed | < 10s | < 5s | < 3s | < 2s | **< 0.5s** |
| Exclusion ratio | < 5% | < 5% | < 3% | < 2% | **< 1%** |
| Test isolation | Optional | Optional | Required | Required | Required |
| Documented strategy | Optional | Optional | Required | Required | Required |
| Full PIT matrix required | No | No | Yes | Yes | Yes |
| Partial PIT cap | Silver | Silver | Silver | Silver | Silver |

**R_direct fallback** (when source analysis is unavailable): R (all killers) thresholds are used with lenient values that accommodate structural transitive kill inflation: Bronze <= 20, Silver <= 12, Gold <= 8, Platinum <= 6, Perfection <= 5.

**Note:** M_t (transitive methods per test) is reported as an informational diagnostic but does NOT affect tier calculation. It was removed as a gate in v3.0 because it measures call-graph structure, not test quality. See Axiom 3 for the rationale.

**Note:** The tier "Diamond" from version 1.0 has been removed. The tier progression is now Bronze -> Silver -> Gold -> Platinum -> Perfection (5 tiers, down from 6). This simplifies the table and removes an intermediate tier that was not meaningfully distinct from Platinum.

### 4.2 Threshold Rationale

**M_p thresholds (primary methods per test):**
- `<= 5` (Bronze): Lenient. Allows tests that directly call up to 5 production methods.
- `<= 4` (Silver): Moderate. Tests must be somewhat focused.
- `<= 3` (Gold): Moderate. Accommodates Gradle plugin integration tests that verify multiple methods.
- `<= 2` (Platinum, Perfection): Tight. Each test exercises at most 2 primary production methods directly. Achievable for pure-function libraries (M_p = 1) and Gradle plugins (M_p = 1-2 with correct source analysis excluding Gradle API calls).

**~~M_t thresholds (transitive methods per test)~~ — REMOVED in v3.0:**
M_t is now informational only. It was removed because it measures call-graph structure, not test quality. For Gradle plugin libraries, the structural minimum M_t is 6 (orchestrator methods like `apply()` transitively reach 6+ sub-methods). Penalizing this conflates good integration testing with mega-testing.

**R_direct thresholds (direct killers per mutation):**
- `<= 10` (Bronze): Very lenient. Allows significant direct redundancy.
- `<= 6` (Silver): Moderately lenient.
- `<= 4` (Gold): Limits redundancy. Each mutation killed by at most 4 tests that directly call the mutated method.
- `<= 3` (Platinum, Perfection): Tight. Each mutation killed by at most 3 tests that directly call the mutated method. This is the practical minimum: 1 primary test + 1-2 secondary tests that also directly call the method for different scenarios.

**R fallback thresholds (all killers, when source analysis unavailable):**
- Bronze <= 20, Silver <= 12, Gold <= 8, Platinum <= 6, Perfection <= 5.
- These are lenient because R (all killers) includes transitive kills, which are structurally inflated for orchestrator-heavy codebases.

**Coverage thresholds:**
- Instruction and branch coverage thresholds are set to be achievable with well-designed unit tests. 100% (Perfection) requires every instruction and branch to be exercised.
- Mutation score thresholds are set relative to coverage. 100% mutation score (Perfection) requires every mutation to be killed, which requires excluding equivalent mutants.

**Full PIT matrix requirement:**
- Bronze and Silver: `fullMutationMatrix` is not required. Gate 3b (R_direct) is evaluated using the fallback R metric with lenient thresholds. Without the full matrix, only one killing test is reported per mutation, and R is always 1 (trivially passing). The fallback R thresholds accommodate this.
- Gold and above: `fullMutationMatrix=true` is REQUIRED. Without it, R_direct cannot be computed accurately and the system uses R with fallback thresholds, which are more lenient. The tier may be capped at Silver if the data quality is insufficient.

**Partial PIT cap:**
- If the PIT run is `partial="true"`, the tier is capped at Silver regardless of other metrics. See Section 3.8.

### 4.3 Tier Calculation Algorithm

```
1. Compute all metrics (Sections 2.1-2.9).
2. Check for partial PIT run (Section 3.8). If partial, set tier_cap = Silver.
3. For each tier from Bronze to Perfection (ascending):
   a. If tier > tier_cap, break.
   b. Check all metrics against tier thresholds.
   c. If ALL metrics meet thresholds, continue to next tier.
   d. If ANY metric fails, the achieved tier is the previous tier. Stop.
4. The achieved tier is the highest tier where all metrics passed.
5. If no tier passes (Bronze fails), the achieved tier is "Below Bronze".
```

### 4.4 Floor Enforcement

The `--floor` argument to `check-tier.sh` specifies the minimum acceptable tier. If the achieved tier is below the floor, the script exits with code 1 (build failure). If the achieved tier is at or above the floor, the script exits with code 0.

Default floor: **Gold**.

## 5. Verification Strategy

This section defines how each metric is computed in CI, what data sources are needed, and how the `check-tier.sh` script should be modified to implement this specification.

### 5.1 Data Sources

| Data Source | Format | Purpose | Required |
|-------------|--------|---------|----------|
| JaCoCo XML | `jacocoTestReport.xml` | Instruction coverage, branch coverage, exclusion ratio | Yes (always) |
| PIT XML | `mutations.xml` | Mutation score, M_p (transitive fast-path), M_t, R, kill matrix | Yes (always) |
| JUnit XML (unit) | `TEST-*.xml` in test results dir | Unit test count, unit test speed | Yes (if unit tests exist) |
| JUnit XML (integration) | `TEST-*.xml` in integration test results dir | Integration test count, integration test speed | No (if no integration tests) |
| Test source code | `*.kt` / `*.java` in test source dirs | M_p primary method analysis (Phase 2) | Yes (if Phase 2 is triggered) |
| Production source code | `*.kt` / `*.java` in main source dirs | M_p primary method classification | Yes (if Phase 2 is triggered) |
| Compiled `.class` files | `build/classes/*/main/**/*.class` | Exclusion ratio computation | Yes (always) |

### 5.2 Required PIT Configuration

The following PIT configuration flags are REQUIRED for full tier evaluation:

```gradle
pitest {
    mutationThreshold = 0  // Do not let PIT fail the build; check-tier.sh handles tiering
    fullMutationMatrix = true  // REQUIRED for Gate 3b (R_direct) at Gold+
    exportLineCoverage = true  // REQUIRED for cross-referencing JaCoCo and PIT
    outputFormats = ['XML']  // XML format for machine parsing
    // Do NOT set maxMutationsPerClass or similar limiting flags (causes partial runs)
    // Do NOT use historyInput/historyOutput for incremental runs (causes partial runs)
}
```

**`fullMutationMatrix = true`:** Causes PIT to record ALL tests that kill each mutation (not just the first). This is required for Gate 3b (R_direct) at Gold+. Without it, only one killing test is reported per mutation, and the system falls back to R with lenient thresholds.

**`exportLineCoverage = true`:** Causes PIT to export line coverage data in the mutation XML. This allows cross-referencing PIT mutations with JaCoCo coverage data, which is useful for diagnosing coverage-only tests (Section 3.9) and zero-mutation methods (Section 3.2).

**`outputFormats = ['XML']`:** The XML format is required for machine parsing. HTML and CSV formats are for human consumption only.

**Flags that MUST NOT be set:**
- `maxMutationsPerClass`: Limits mutations per class -> partial run.
- `maxMutations`: Limits total mutations -> partial run.
- `historyInput` / `historyOutput`: Incremental mode -> partial run.
- `mutators` must include the full default set (do not reduce to fewer mutators to game the mutation score).

### 5.3 Required JaCoCo Configuration

```gradle
jacocoTestReport {
    xml.required = true
    csv.required = false
    html.required = true  // for human review
    // Exclusions must be documented in TEST_STRATEGY.md
    // Do NOT exclude broad packages to game coverage
}
```

### 5.4 check-tier.sh Modifications

The current `check-tier.sh` script (as of version 1.0) computes:
- Instruction coverage, branch coverage, mutation score (correct, retain).
- TES (Test Efficiency Score) — REMOVE. Replaced by M_p, M_t, and R.
- Zero-kill ratio — REMOVE. Subsumed by mutation score and R.
- Exclusion ratio (correct, retain).
- Test speed (correct, retain).
- Test isolation, documented strategy (correct, retain).

**Required modifications:**

1. **Remove TES computation** (lines computing `tes`, `zero_kill_count`, `bloat_tests`, `kt_mean`, `kt_variance`, `kt_factor`, `kt_accuracy`, `kt_precision`). Remove all TES-related output.

2. **Add M_t computation (Gate 3b):**
   - For each killed mutation, read `mutatedClass` and `mutatedMethod`.
   - For each test in `killingTests`, add `(mutatedClass, mutatedMethod)` to the test's method set.
   - `M_t(t) = |method_set(t)|` (count of distinct methods).
   - `M_t = max_t M_t(t)`.
   - Compare against `M_t_threshold` for each tier.

3. **Add M_p computation (Gate 3a):**
   - Phase 1 (PIT fast-path): For each test, `M_t(t)` is already computed (from step 2). If `M_t(t) <= M_p_threshold` for all tests, PASS. No source analysis needed.
   - Phase 2 (source analysis): For tests where `M_t(t) > M_p_threshold`, parse the test source file. Identify direct production method calls. Compute `M_p(t)`. Use `M_p(t)` for the gate.
   - Source analysis implementation: Use a Python script that:
     - Locates the test source file by searching test source directories for the test method name.
     - Parses the test method body (regex for Kotlin/Java method calls).
     - Classifies each call as production / test framework / assertion / test utility / stdlib.
     - Counts production method calls as primary methods.
   - Requires: test source directory path (configurable, default `src/test/kotlin/`).

4. **Add R computation (Gate 3b (fallback)):**
   - For each killed mutation, parse `killingTests` (pipe-separated, `[method:name()]` format).
   - Count distinct test names -> `R(m)`.
   - `R = max(R(m))` over all killed mutations.
   - Requires: `fullMutationMatrix=true` in PIT config. If `killingTests` is absent and only `killingTest` (singular) is present, emit a WARNING and set `R = N/A` (gate fails at Gold+).

5. **Add partial PIT detection:**
   - Parse the PIT XML root element for `partial="true"`.
   - If partial, emit WARNING and cap tier at Silver.

6. **Update tier table:**
   - Replace TES row with M_p, M_t, and R rows.
   - Remove Diamond tier (5 tiers now: Bronze, Silver, Gold, Platinum, Perfection).
   - Add "Full PIT matrix required" row (No for Bronze/Silver, Yes for Gold+).
   - Add "Partial PIT cap" row (Silver for all tiers).

7. **Add `--test-src-dir` argument:**
   - New argument to specify the test source directory for Phase 2 source analysis.
   - Default: `src/test/kotlin` (Kotlin) with fallback to `src/test/java` (Java).

8. **Add `--main-src-dir` argument:**
   - New argument to specify the production source directory for Phase 2 method classification.
   - Default: `src/main/kotlin` with fallback to `src/main/java`.

9. **Update output format:**
   - Remove TES, K/T, zero-kill, subset bloat lines.
   - Add M_p, M_t, and R lines:
     ```
     M_p (granularity):  3 (max primary methods per test)  [threshold: 2]
     M_t (transitivity): 5 (max transitive methods per test) [threshold: 4]
     R (redundancy):     2 (max killers per mutation)       [threshold: 5]
     ```
   - If M_p was computed via PIT fast-path only: add note `[PIT transitive — source analysis not triggered]`.
   - If M_p required source analysis: add note `[source analysis: N tests analyzed]`.
   - If M_p source analysis failed for any test: add note `[source analysis: N tests failed, using transitive fallback]`.

### 5.5 Source-Code Analysis for Primary M_p

The source-code analysis for primary M_p (Phase 2) is the most complex new component. It must:

1. **Locate the test source file:**
   - Given a test method name (e.g., `testCheckout`), search all `.kt` and `.java` files in the test source directory.
   - Match the method name in the file content (regex: `fun \`testCheckout\`()` for Kotlin, `void testCheckout()` for Java).
   - If multiple files match, use the first match (PIT test names should be unique within a project).

2. **Extract the test method body:**
   - For Kotlin: Find `fun \`methodName\`(...)` followed by `{`. Track brace depth to find the closing `}`.
   - For Java: Find `void methodName()` (or `@Test public void methodName()`) followed by `{`. Track brace depth.
   - Handle annotations (`@Test`, `@DisplayName`, etc.) preceding the method.

3. **Identify direct method calls:**
   - Parse the method body for call expressions: `identifier(args)` or `receiver.identifier(args)`.
   - For Kotlin: regex `\b([a-zA-Z_][a-zA-Z0-9_]*)\s*\(` to find call sites. Filter out language keywords (`if`, `when`, `for`, `while`, `return`, etc.).
   - For each call site, determine the callee:
     - If the callee is an assertion method (`assertEquals`, `assertTrue`, `assertNotNull`, `verify`, `expect`, etc.) -> NOT a primary method.
     - If the callee is a test framework method (`setUp`, `tearDown`, `beforeEach`, `afterEach`, etc.) -> NOT a primary method.
     - If the callee is defined in the same test file (a test helper method) -> NOT a primary method.
     - If the callee is a Kotlin/Java stdlib method (`println`, `listOf`, `mapOf`, `setOf`, etc.) -> NOT a primary method.
     - If the callee is a production method (defined in a production source file) -> **primary method**. Count it.
   - Deduplicate primary method names (calling `foo.bar()` twice counts as 1 primary method: `bar`).

4. **Classify a callee as production vs non-production:**
   - Build an index of all method names defined in production source files (`src/main/kotlin/**/*.kt`, `src/main/java/**/*.java`).
   - If the callee name matches a method in the production index -> production method.
   - If the callee name matches a method in the test source files -> test utility method.
   - If the callee name matches a known assertion / framework method -> framework method.
   - If the callee name is not found in any source file -> likely a stdlib / JDK method -> NOT a primary method.
   - Ambiguity: If a method name exists in both production and test sources, it is classified as production (conservative — may slightly overcount M_p, which is the safe direction for a granularity gate).

5. **Output:**
   - For each test analyzed: `M_p(t) = |{distinct primary methods}|`.
   - `M_p = max_t M_p(t)`.

**Performance consideration:** Source analysis is only triggered for tests where `M_t(t) > M_p_threshold`. In practice, this is a small fraction of tests (most tests have low transitive M_t). The analysis is O(test_method_body_size) per test, which is fast.

**Limitation:** The regex-based approach cannot handle all Kotlin language features (extension functions, operator overloading, infix functions, string templates with `${}`). For these cases, the analysis may miss some primary method calls (undercounting M_p) or include some non-primary calls (overcounting). The conservative direction is overcounting (false positive = gate fails when it should pass), which is safe because it triggers human review. A future implementation using Kotlin compiler PSI would eliminate this limitation.

- `testFacade()` — calls `Facade.run()`. `Facade.run()` calls `A.doA()`, `B.doB()`, `C.doC()`. All 4 methods' mutations killed.
- M_p = 1. <= 1 PASS.
- M_t = 4. > 3 -> FAIL Gate 3b. The facade test reaches too many methods.
- The fix: mock `A`, `B`, `C` in `testFacade`, reducing M_t to 1.

**Failing example (R blocks Platinum):**

Test suite: 1 mutation in `B.bar()` is killed by 4 tests:
- `testB` (direct test for `B.bar()`).
- `testA` (test for `A.foo()` which calls `B.bar()` — transitive kill).
- `testIntegration` (integration test exercising `A.foo()` and `B.bar()`).
- `testProperty` (property-based test exercising `B.bar()`).
- R(m) = 4. > 3 -> FAIL Gate 3b (fallback). Too many tests kill the same mutation.
- The fix: remove `testProperty` or mock `B.bar()` in `testA` and `testIntegration`.

### 6.5 Perfection (Instr = 100%, Branch = 100%, Mut = 100%, M_p <= 1, M_t <= 3, R <= 3)

**Passing example (theoretical):**

Production code: N classes, each with 1 method. No inter-class calls (all methods are leaf methods — they do not call other production methods). Each class has its own dedicated test. All dependencies (if any) are mocked.

Test suite: N tests, each calling exactly 1 production method directly.
- Each test: M_p = 1, M_t = 1 (no transitive calls since all methods are leaf methods).
- R = 1 (each mutation killed by exactly 1 test; no transitive kills since no inter-class calls).
- All mutations killed (no equivalent mutants; all mutatable code is killable).
- All instructions and branches covered.

Metrics:
- Instruction: 100%. = 100% PASS.
- Branch: 100%. = 100% PASS.
- Mutation: 100%. = 100% PASS.
- M_p = 1. <= 1 PASS.
- M_t = 1. <= 3 PASS.
- R = 1. <= 3 PASS.
- Exclusion ratio: 0% (nothing excluded). < 1% PASS.
- All gates pass at Perfection.

**This is achievable only for codebases where:**
1. No production method calls another production method (all methods are leaf methods, or all inter-method calls are mocked in tests).
2. No equivalent mutants exist (or all are excluded from PIT, with exclusion ratio < 1%).
3. Every instruction and branch is covered.
4. Every mutation is killed.
5. Each test exercises exactly 1 primary method.
6. Each test's transitive footprint is at most 3 methods.
7. Each mutation is killed by at most 3 tests.

See Section 7 for the full analysis of Perfection reachability.

**Failing example (should be blocked at Perfection):**

Any of the following conditions block Perfection:
- A single uncovered instruction (instruction < 100%).
- A single untaken branch (branch < 100%).
- A single surviving mutation (mutation < 100%).
- A single test that calls 2 production methods directly (M_p > 1).
- A single test that transitively reaches 4+ methods (M_t > 3).
- A single mutation killed by 4+ tests (R > 3).
- An equivalent mutant that is not excluded (mutation < 100%).
- An exclusion ratio >= 1% (too much code excluded).

## 7. Perfection: Definition and Reachability

### 7.1 What Does a Perfection-Tier Suite Look Like?

A Perfection-tier test suite satisfies ALL of the following simultaneously:

1. **100% instruction coverage:** Every JVM bytecode instruction in the instrumented production codebase is executed by at least one test.
2. **100% branch coverage:** Every branch edge (true and false direction of every conditional) is traversed by at least one test.
3. **100% mutation score:** Every mutation generated by PIT is killed by at least one test. No mutations survive. No mutations have NO_COVERAGE. No equivalent mutants are present (they must be excluded from PIT via configuration).
4. **M_p <= 2:** Each test directly calls at most 2 production methods. Tests calling 3+ production methods directly are mega-tests.
5. **R_direct <= 3:** Each mutation is killed by at most 3 tests that directly call the mutated method. Transitive kills (from orchestrator tests) are NOT counted.
6. **Exclusion ratio < 1%:** Less than 1% of production bytecode is excluded from coverage measurement.
7. **Test isolation verified:** Tests pass with randomized execution order.
8. **Documented strategy:** `TEST_STRATEGY.md` exists and documents the testing approach.
9. **Full PIT matrix:** `fullMutationMatrix=true` is configured. The PIT run is not partial.

**Note:** M_t is NOT a Perfection requirement in v3.0. A Perfection-tier suite may have M_t = 6+ for orchestrator tests. This is expected and acceptable for Gradle plugin libraries and other orchestrator-heavy architectures.

**Structural properties of a Perfection-tier suite:**
- Each test **mocks external dependencies** where practical, but does NOT need to mock internal orchestrator methods (M_t is no longer gated).
- Each test exercises at most **2 primary production methods** directly (M_p <= 2).
- Each mutation is killed by at most **3 tests that directly call the mutated method** (R_direct <= 3). Transitive kills (from orchestrator tests) are NOT counted.
- Equivalent mutants are **identified and excluded** from PIT — this makes 100% mutation score achievable.
- The test suite has **no directly redundant tests** — at most 3 tests directly test any single method.
- M_t may be high (6+) for orchestrator tests — this is acceptable and expected for Gradle plugin libraries. It is reported as information, not gated.

### 7.2 Is Perfection Reachable for a Gradle Plugin Library?

**Short answer (v3.0): YES.** With M_t removed as a gate and R replaced by R_direct, Perfection is structurally achievable for a Gradle plugin library WITHOUT production code refactoring.

**Analysis (v3.0):**

The v2.0 spec concluded that Perfection was "practically very difficult" for Gradle plugin libraries because:
1. M_t <= 3 was unreachable for orchestrator methods like `apply()` (structural minimum: 6)
2. R <= 3 was unreachable for shared utility methods (structural minimum: 3-5 due to transitive kills)

Both barriers are eliminated in v3.0:
1. **M_t is no longer a gate.** Orchestrator tests with M_t = 6+ no longer block any tier.
2. **R_direct replaces R.** Transitive kills (which inflated R to 3-5 for shared methods) are excluded. R_direct for a shared utility method is typically 1 (only the direct test counts).

**Remaining barriers to Perfection for a Gradle plugin library:**

| Barrier | Impact | Mitigation |
|---------|--------|------------|
| ~~Inter-method calls inflate M_t~~ | ~~M_t > 3~~ | **RESOLVED: M_t removed as gate in v3.0** |
| ~~Transitive kills inflate R~~ | ~~R > 3~~ | **RESOLVED: R_direct replaces R in v3.0** |
| Equivalent mutants in data class methods | Mutation score < 100% | Exclude equivalent mutants from PIT; keep exclusion ratio < 1% |
| Surviving mutations | Mutation score < 100% | Kill or exclude all surviving mutations |
| M_p > 2 (tests calling 3+ primary methods) | Blocks Platinum/Perfection | Fix source analysis to exclude Gradle API calls; break tests that genuinely call 3+ production methods |
| Unit test speed > 10ms | Blocks Perfection | Structural for ProjectBuilder-based tests; may require test class consolidation |

**Verdict (v3.0):** Perfection is **structurally reachable** for a Gradle plugin library if and only if:
1. ~~M_t is manageable~~ — N/A, M_t is not a gate.
2. All equivalent mutants are identified and excluded from PIT, with the exclusion ratio remaining below 1%.
3. Every instruction, branch, and mutation is covered/killed (mutation score = 100%).
4. Each test exercises at most 2 primary methods (M_p <= 2). With correct source analysis excluding Gradle API calls, this is achievable.
5. Each mutation is killed by at most 3 tests that directly call the mutated method (R_direct <= 3). With transitive kills excluded, this is achievable.

**This does NOT require production code refactoring.** The developer needs to ensure tests are focused (M_p <= 2) and not directly redundant (R_direct <= 3), and that all mutations are killed or properly excluded. These are test quality concerns, not production code architecture concerns.

### 7.3 Practical Ceiling for a Gradle Plugin Library

For most real-world Gradle plugin libraries (v3.0), the practical ceiling is:

- **Gold** is achievable with moderate effort: 90%+ instruction coverage, 85%+ branch coverage, 85%+ mutation score, M_p <= 3, R_direct <= 4. M_t is not a gate.
- **Platinum** is achievable with focused effort: 95%+ instruction coverage, 90%+ branch coverage, 95%+ mutation score, M_p <= 2, R_direct <= 3. Requires well-focused tests and correct M_p source analysis (excluding Gradle API calls).
- **Perfection** is achievable when all mutations are killed or properly excluded: 100% coverage, 100% mutation score, M_p <= 2, R_direct <= 3, exclusion ratio < 1%, unit test speed < 10ms. The speed threshold may be the hardest barrier for ProjectBuilder-based test suites.

The tier system is designed so that Gold is the default floor (Section 4.4), Platinum is the stretch goal, and Perfection is achievable for well-tested codebases of ALL types — including Gradle plugin libraries — without production code refactoring.

## 8. Known Limitations

This section documents the known limitations of the Testing Tier System honestly and completely. Each limitation includes its impact and the recommended mitigation.

### 8.1 M_t Undercounts When Methods Have 0 Mutations

**Limitation:** `M_t` counts only methods whose mutations are killed by the test. If a test transitively reaches a method that has 0 mutations (e.g., a method in `avoidCallsTo`, a method with no mutatable bytecode, or a method excluded from PIT), that method is invisible to `M_t`. This means `M_t` may undercount the true transitive reach of a test.

**Impact:** A test that transitively reaches 10 methods but only 3 of them have mutations will have `M_t = 3`, not 10. The test appears finer-grained than it actually is. This could allow a facade mega-test to pass Gate 3b if the transitive methods happen to have few mutations.

**Mitigation:** The exclusion ratio (Section 2.9) tracks how much code is excluded from PIT. If the exclusion ratio is low (< 1% at Perfection), the undercount is minimal. The `check-tier.sh` script's warning for "classes with branches but no mutations" helps identify methods with 0 mutations. The project should document in `TEST_STRATEGY.md` which methods have 0 mutations and why.

**Unresolved:** There is no way to compute the true transitive reach of a test (including methods with 0 mutations) from PIT data alone. This would require call-graph analysis from the test's bytecode, which is a future enhancement.

### 8.2 R Inflated by Transitive Kills

**Limitation:** When production method `A.foo()` calls `B.bar()`, a test for `A.foo()` will transitively execute `B.bar()`. If `B.bar()` has mutations, and the test for `A.foo()` kills them, then `R(m) >= 2` for `B.bar()`'s mutations (killed by both `testA` and `testB`). This inflates `R` for `B.bar()`'s mutations.

**Impact:** The Perfection tier requires `R <= 3` (not `R <= 1` as in version 1.0). This accommodates the common pattern of 1 direct test + 1 transitive killer + 1 integration test = 3 killers. However, if a method is called by many other methods that all have their own tests, `R` can exceed 3 even with good test design.

**Mitigation:** The primary mitigation is mocking: in `testA`, mock `B.bar()` so that `testA` does not kill `B.bar()`'s mutations. This requires `testA` to test `A.foo()` in isolation, with all dependencies mocked. This is the "unit test ideal" — each test tests exactly one method with all dependencies mocked.

**Acceptance:** The system accepts that `R <= 3` is the practical minimum for non-trivial codebases. The tier table is designed so that Platinum and Perfection both require `R <= 3`, which is achievable with disciplined mocking.

**No transitive kill discount:** The system does NOT discount transitive kills from `R`. If `testA` kills `B.bar()`'s mutations transitively, it counts as a killer. The rationale is that the redundancy is real — if `testB` is removed, `testA` still kills `B.bar()`'s mutations, so `testB` is redundant for those mutations. The fix is to mock `B.bar()` in `testA` or to accept the redundancy.

### 8.3 Coverage-Only Tests Are Invisible to Axiom 3 Gates

**Limitation:** A test that exercises production code (contributing to coverage) but does not assert on the behavior (and thus kills no mutations) is invisible to all three Axiom 3 gates (M_p, M_t, and R). The test does not appear in any mutation's `killingTests` list, so it does not contribute to `M_p`, `M_t`, or `R`.

**Impact:** A test suite could have many coverage-only tests that inflate the test count (lowering test speed) without contributing to mutation score or being detected by Axiom 3 gates. The only defense is Axiom 2 (mutation score) — if the test suite has many coverage-only tests, the mutation score will be low because those tests don't kill mutations.

**Mitigation:** The mutation score (Axiom 2) is the primary defense. A high coverage / low mutation score gap indicates coverage-only tests. The `check-tier.sh` script should emit a diagnostic when `instruction_coverage - mutation_score > 25%` (coverage is much higher than mutation score, suggesting coverage-only tests). This diagnostic is informational, not a gate failure.

**Unresolved:** There is no way to detect coverage-only tests from PIT data alone (PIT only records killed mutations, not exercised-but-not-killed mutations). JaCoCo coverage data could be cross-referenced with PIT kill data to identify tests that exercise code but don't kill mutations, but this requires per-test coverage data (not just aggregate coverage), which JaCoCo does not provide by default.

### 8.4 Equivalent Mutants

**Limitation:** Some mutations generated by PIT are **equivalent** — the mutated code is semantically equivalent to the original, and no test can kill it. These mutations will always have status `SURVIVED`, making 100% mutation score (Axiom 2) unreachable.

**Examples:**
- `i + 1` -> `i - 1` is killable. But `i * 1` -> `i * -1` may be equivalent if `i` is always 0.
- Replacing `return true` with `return false` is killable. But replacing `a = b` with `b = a` may be equivalent for symmetric operations.
- Removing a `++` on a variable that is never read afterward is equivalent.

**Impact:** The Perfection tier (100% mutation score) is unreachable for any codebase with equivalent mutants. This is a fundamental limitation of mutation testing.

**Mitigation:** Equivalent mutants must be identified and excluded from PIT via configuration (`excludedMutators` or `excludedClasses`). The exclusion ratio (Section 2.9) tracks how much code is excluded. The project should document equivalent mutants in `TEST_STRATEGY.md`.

**Unresolved:** There is no automated way to detect equivalent mutants with certainty. Automated equivalence detection is an open research problem. The system relies on human identification and exclusion.

### 8.5 Partial PIT Run Issue

**Limitation:** If PIT is interrupted or run with limiting flags, the PIT XML is marked `partial="true"` and the mutation data is incomplete. The mutation score, M_p, M_t, and R are all unreliable.

**Impact:** A partial run can produce a misleadingly high mutation score (if the mutations that were tested happen to be mostly killed) or a misleadingly low one (if PIT was interrupted early). The M_p, M_t, and R values are also unreliable because the kill matrix is incomplete.

**Mitigation:** The `check-tier.sh` script detects `partial="true"` and caps the tier at Silver (Section 3.8). The project must re-run PIT with a full mutation matrix to certify Gold or above.

**Unresolved:** There is no way to compute reliable mutation metrics from a partial run. The only solution is a full re-run, which can be expensive for large codebases (PIT is slow). Incremental mutation testing (using `--historyInput` / `--historyOutput`) can speed up re-runs but still produces partial XML for the incremental run.

### 8.6 Source Analysis Limitations (Phase 2)

**Limitation:** The source-code analysis for primary M_p (Phase 2, Section 5.5) uses regex-based parsing of Kotlin/Java source files. This approach cannot handle all language features:

- **Extension functions:** `fun String.myExtension()` is called as `"hello".myExtension()`. The regex may not identify `myExtension` as a call.
- **Operator overloading:** `a + b` calls `a.plus(b)`. The regex sees `+`, not `plus`.
- **Infix functions:** `a infix b` calls `a.infix(b)`. The regex sees `a` and `b` as separate identifiers.
- **String templates:** `"${obj.method()}"` calls `method()` inside a string. The regex may or may not catch this depending on the pattern.
- **Implicit receivers:** In Kotlin, `foo()` inside a class calls `this.foo()`. The regex catches `foo()` but may misclassify it as a stdlib call if `foo` is not found in the production index.
- **Destructuring declarations:** `val (a, b) = pair` calls `pair.component1()` and `pair.component2()`. Not caught by regex.

**Impact:** The source analysis may undercount or overcount primary methods. Undercounting is safe (the gate passes when it should fail — false negative). Overcounting is conservative (the gate fails when it should pass — false positive, triggering human review).

**Mitigation:** The conservative direction is overcounting (classifying ambiguous calls as primary methods). This is safe because it triggers human review. A future implementation using Kotlin compiler PSI or bytecode analysis would eliminate this limitation.

**Unresolved:** Regex-based parsing is inherently limited. The only complete solution is compiler-level analysis (PSI for Kotlin, AST for Java) or bytecode analysis (parsing the test's `.class` file to find direct `INVOKEVIRTUAL` / `INVOKESTATIC` calls to production methods). Both require significant implementation effort.

### 8.7 M_p, M_t, and R Do Not Capture All Forms of Test Waste

**Limitation:** M_p (granularity), M_t (transitivity), and R (redundancy) capture three specific forms of test waste: mega-tests, facade mega-tests, and redundant tests. They do not capture:

- **Slow tests:** A test that takes 10 seconds but kills unique mutations and exercises 1 primary method passes all Axiom 3 gates. It is penalized only by the test speed metric.
- **Brittle tests:** A test that fails when unrelated code changes (e.g., a test that depends on specific string formatting). Not detectable from PIT or JaCoCo data.
- **Assertion-free tests:** A test that calls production code but has no assertions. These are coverage-only tests (Section 3.9) and are invisible to Axiom 3.
- **Over-mocking:** A test that mocks so much that it tests nothing meaningful. These tests may kill mutations (if the mock is configured to detect unexpected calls) but provide little value. Not detectable from PIT data.

**Impact:** A test suite could satisfy all three axioms and still have quality issues not captured by the tier system.

**Mitigation:** The tier system is a necessary but not sufficient condition for test quality. Code review, test strategy documentation, and human judgment remain essential. The tier system provides a quantitative floor, not a qualitative ceiling.

## 9. Adversary Findings — Resolution Log

This section documents the adversary's findings from the previous review round (CRITICAL-1 through CRITICAL-7) and how each was resolved in this specification.

### CRITICAL-1: M_p from PIT Is Transitive, Not Primary

**Finding:** The version 1.0 spec used PIT data to compute `M_p`, but PIT data gives the transitive method count, not the primary method count. This means `M_p` was overcounting, flagging fine-grained tests as mega-tests.

**Resolution:** The spec now has two separate metrics:
- `M_p` (Gate 3a): Primary methods per test, computed via hybrid PIT fast-path + source analysis (Section 2.4.3). The PIT fast-path uses `M_t` as an upper bound; if it passes, no source analysis is needed. If it fails, source analysis computes the true primary count.
- `M_t` (Gate 3b): Transitive methods per test, computed directly from PIT data (Section 2.5). This is the metric that was previously conflated with `M_p`.

The two metrics serve different purposes: `M_p` catches direct mega-tests, `M_t` catches facade mega-tests. Both are needed (Section 1, Gate 3a and Gate 3b).

### CRITICAL-2: R = 1 at Perfection Is Structurally Unreachable

**Finding:** The version 1.0 spec required `R <= 1` at Perfection (each mutation killed by exactly 1 test). This is unreachable for any non-trivial codebase where methods call other methods, because transitive kills inflate `R` to >= 2.

**Resolution:** The Perfection tier now requires `R <= 3` (Section 4.1, Section 2.6.4). This accommodates the common pattern of 1 direct test + 1 transitive killer + 1 integration test = 3 killers. `R <= 3` is tight but achievable. It still blocks egregious redundancy (e.g., 10 tests killing the same mutation). See Section 7 for the full analysis.

### CRITICAL-3: Facade Mega-Tests Pass M_p But Are Still Mega-Tests

**Finding:** A test that calls 1 production method (passing `M_p <= 1`) but that method is a facade calling 50 other methods is still a mega-test. The version 1.0 spec had no gate to catch this.

**Resolution (v2.0):** The M_t gate (Gate 3b) was added to catch facade mega-tests. M_t counts the transitive closure of methods whose mutations are killed by the test.

**Update (v3.0):** M_t was subsequently REMOVED as a gate because it conflated orchestrator integration testing with mega-testing. See Axiom 3 for the full rationale. The facade mega-test scenario is now caught by M_p alone — a test that calls 1 facade entry point which fans out to 50 methods is a legitimate integration test for that entry point, not a mega-test.

### CRITICAL-4: Synthetic and Lambda Methods Not Formally Classified

**Finding:** The version 1.0 spec did not formally define how synthetic methods (Kotlin data class generated methods, lambda bodies, companion object methods) are treated in `M_p` and `M_t`.

**Resolution:** Section 3.5 now formally classifies synthetic methods:
- **M_p (Gate 3a):** NOT counted as primary methods in source analysis (they are not directly called from test source).
- **M_t (Gate 3b):** Counted if they have mutations in PIT (they contribute to the transitive footprint).
- **R (Gate 3b (fallback)):** Counted normally.
- **Axiom 1 and 2:** Fully in scope.

Data class getters have a specific formal rule (Section 3.5, "Data class getters — formal rule").

### CRITICAL-5: Source Analysis Failure Mode Undefined

**Finding:** The version 1.0 spec did not define what happens when source analysis (Phase 2) fails — e.g., the test source file cannot be found, or the method body cannot be parsed.

**Resolution:** Section 2.4.4 now defines the fallback:
- If source analysis fails, `M_p(t) = M_t(t)` (use the transitive upper bound as conservative fallback).
- The `check-tier.sh` script MUST emit a WARNING when the fallback is used.
- Classification ambiguity defaults to "production method" (conservative overcount).

### CRITICAL-6: Exclusion Ratio Gate Not Implemented

**Finding:** The version 1.0 spec defined the exclusion ratio but the `check-tier.sh` script did not fully implement it as a gate at all tiers.

**Resolution:** Section 2.9 now includes an explicit implementation requirement: the exclusion ratio gate MUST be implemented in `check-tier.sh` and MUST be evaluated at all tiers. The implementation must walk the compiled class output directory, count bytecode instructions of excluded classes using `javap -c`, and compare against the tier threshold.

### CRITICAL-7: Coverage-Only Tests Invisible to Axiom 3

**Finding:** The version 1.0 spec noted that coverage-only tests (tests that exercise code but kill no mutations) are invisible to the Axiom 3 gates, but did not formally define the edge case or the diagnostic.

**Resolution:** Section 3.9 now formally defines coverage-only tests, their effect on each gate, and the canonical handling. A diagnostic is specified: emit when `instruction_coverage - mutation_score > 25%`. This is informational, not a gate failure. The mutation score (Axiom 2) is the primary defense.
